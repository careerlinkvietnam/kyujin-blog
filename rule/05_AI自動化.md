# AI記事自動生成・投稿 業務マニュアル
## Ver 5.0 (Site-Wide Growth Edition / Claude Code & WP REST API Ready)

---

## 1. 業務の目的

「世界最高の東南アジア就職サイト」を目指し、
AIによるコンテンツ生成・改善・淘汰を **自律的に回し続ける運用仕様** を定義する。

### Ver 5.0 の重要な変更点

**「記事生成」から「サイト全体最適化」へ**

本システムはトラフィック増ではなく **CV（会員登録・応募）増** を北極星とする。
記事生成は導線整備・技術SEO改善の**後**に実行する。

---

## 2. 運用の3原則（更新）

1. **CV-First**
   - トラフィックがあってもCVに繋がらなければ価値ゼロ
   - 既存高トラフィックページの導線整備を最優先

2. **Diagnosis-Driven**
   - 週次のサイト診断結果に基づいて行動を決定
   - 「なんとなく記事を書く」を禁止

3. **Metabolism**
   - 価値のない記事を捨て、サイト全体のドメインパワーを維持

### Step 0: 週次サイト診断（必須起点）

**実行タイミング**: 週に1回（月曜日）、または新規セッション開始時

**実行内容**:
「【実行用】サイト全体診断マニュアル.md」に従い、以下を診断：

1. トラフィック構造診断（転職関連 vs 非転職関連）
2. ページ別導線診断（CTA有無）
3. コンテンツ充実度診断（キーワードカバレッジ）
4. 技術SEO診断

**出力**:
- `診断レポート_YYYY-MM-DD.md`
- `診断タスク_YYYY-MM-DD.json`

**診断をスキップしてはならない**:
診断なしに記事生成を開始することを禁止する。

### Step 1: タスク選定と重複チェック

`プロジェクト進捗履歴.md` を読み込み、以下の優先順位で処理する。

1. **Priority S** - 導線整備（CTA注入、内部リンク追加）
2. **Priority A** - 技術SEO、リライト
3. **Priority B** - CTR改善、メタタグ最適化
4. **Priority C** - 新規記事作成
5. **Priority D** - Type D（Industry Report）月1本

#### 重複防止

- 同一検索意図の記事が既に存在する場合、新規作成しない
- 「重複スキップ」として履歴に記録し、処理を終了する

---

### Step 2: データ分析と出典管理（競合出典禁止）

#### 1. 自社データ（断定可）

- `jobs_data.json` に基づく数値のみ断定表現を許可する

#### 2. 外部データ（必須ルール｜出典制限あり）

- 対象：生活費・税金・ビザ等
- 使用する場合は必ず本文内に以下を明記：
  - 出典元名称
  - URL
  - 参照年月

##### 使用可能な出典（Allowlist）

- 各国政府・公的機関（移民局、税務当局、統計局等）
- 国際機関（World Bank / IMF / OECD / ILO / ASEAN）
- Big4・国際法律事務所（制度・法令解説に限る）
- 公開指数・データベース（Numbeo 等）
  ※必ず「推定値」である旨を明記

##### 使用禁止の出典

- 競合人材紹介会社・求人サイト・転職メディア
- 個人ブログ、note、SNS投稿
- 出典元・更新日が不明な情報

- 出典条件を満たさない数値は断定しない
- もしくは数値自体を出さず、定性的説明に留める

---

### Step 3: 執筆と内部リンク設計

`コンテンツ戦略ガイド.md`（Type A / B / D）
`ブランドボイス定義書.md`（Mode A / B）
に厳密に従って執筆する。

#### 内部リンク自動実装ルール（必須）

1. **親記事（Pillar）**
   - 国別トップガイド、または就職完全ガイドへリンク
2. **兄弟記事（Related）**
   - 同カテゴリ上位2本へリンク
3. **求人ページへの誘導**
   - 必ず該当国の求人一覧へリンク
4. **CTA必須**
   - 記事末尾に必ず会員登録CTAを設置
   - URL: `https://kyujin.careerlink.asia/register`

---

### Step 4: 品質チェック（Editor Mode）

以下をすべて満たすこと。

1. **禁止表現チェック**
   - 「夢」「グローバル人材」「ご紹介できます」
   - 「絶対」「必ず」
2. **E-E-A-Tチェック**
   - 数値は自社データまたは出典明記済み外部データか
3. **構造チェック**
   - 内部リンクが設置されているか（最低3件）
   - ターゲットに合致した CTA があるか
   - 会員登録URLが正しいか（/register）
4. **導線チェック（NEW）**
   - 求人ページへのリンクがあるか
   - 関連記事へのリンクがあるか

NGの場合は最大2回までリライトする。

### Step 5: WordPress投稿と安全装置（REST API）

WordPress への投稿は **WordPress REST API** を使用して行う。
管理画面からの手動投稿・自動公開は禁止する。

#### 投稿方式

- APIエンドポイント
  - `/wp-json/wp/v2/posts`
- 認証方式
  - アプリケーションパスワード（Basic Auth）
- 投稿形式
  - JSON
- 投稿ステータス
  - `draft`（下書き）固定 → 人間承認後に `publish`

#### 投稿前チェック（必須）

1. **Slug重複チェック**
   - REST API を使用し、同一 slug が既に存在しないか確認
   - 存在する場合は投稿せず処理を中断する

#### 投稿実行

- `POST /wp-json/wp/v2/posts`
- 必須フィールド例：
  - `title`
  - `content`
  - `slug`
  - `status: draft`
  - `categories`
  - `tags`
  - `meta`（必要な場合）

#### 成功判定（厳守）

- HTTP ステータスコード **`201 (Created)`** のみ成功扱い
- レスポンス JSON から以下を取得・保存する：
  - `id`（投稿ID）
  - `link`（記事URL）
  - `slug`

#### 失敗時の扱い

- HTTP 201 以外、タイムアウト、JSON不正は **失敗扱い**
- 失敗時は **次の Step 6 に進まない**
- エラー内容をログに記録して終了する

#### セキュリティルール

- 認証情報（ユーザー名・アプリケーションパスワード）は
  `プロジェクト進捗履歴.md` から取得
- コード・ログ・会話に出力しない

---

### Step 6: 進捗履歴の更新（Transactional Update）

**投稿成功時のみ** 実行する。

1. 該当タスクを `[x]` に変更
2. 以下を記録：
   - 記事ID
   - Slug
   - 主要ターゲットキーワード（GSC分析用）
   - 投稿日時

---

### Step 7: 完了報告

- 成功 / スキップ / エラーを明示して終了する
- 次に実行すべきタスクを提案する

---

## 5. 非交渉ルール

- 競合に権威を渡さない
- 怪しい数字は出さない
- 迷ったら出さない
- 勝ち筋に集中し、負け筋は捨てる
- **診断なしに記事を書かない**
- **CTAなしの記事を公開しない**
