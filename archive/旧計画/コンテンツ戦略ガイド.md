# コンテンツ戦略ガイド Ver.3.0
## CareerLink Asia｜AI自動生成のための記事設計図・変数定義・運用基準（最終運用版）

**適用範囲**：kyujin.careerlink.asia（求人）／ブログ（WordPress）に掲載する全コンテンツ（AI生成・人間執筆を問わない）

---

## 0. このガイドの目的（KPIの紐づけ）
本ガイドは「記事を量産するため」ではなく、以下を同時に満たすための運用基準である。

- **流入最大化**：検索意図に合致した網羅性と内部リンク構造で、指名なし流入を増やす
- **CV最大化**：記事テーマに整合した求人閲覧・応募・相談への導線を設計する
- **信用最大化**：数値・事例・制度情報で事故を起こさず、企業と求職者双方の信頼を積み上げる

---

## 1. コンテンツ生成の絶対原則（Must）

### 1.1 データ・ファースト（ただし「自社データ偏重」で破綻させない）
1) **求人・給与・要件に関する数値根拠は原則 `jobs_data.json`**（自社一次データ）を使う。
2) ただし、**生活費・税・ビザ等**は自社求人データだけでは成立しないため、以下の「許可された外部データ」を利用してよい。

**許可された外部データ（Allowed External Data）**
- 公式統計・官公庁・制度ページ（国別）
- 大手会計・法律事務所の制度解説（国別）
- 物価・生活費の公開指数（出典・更新日が明確なもの）

**必須**：外部データを使う場合、記事中に「出典名」と「更新日（または参照日）」を明記する。

> 禁止：出典不明のブログ・SNS・掲示板の数値を、事実として断定すること

### 1.2 ブランドボイスの適用（Modeの先決）
執筆前に `brand_voice.md` を読み込み、必ず Mode を確定する。
- 求職者向け：**Mode A（Strict Mentor）**
- 企業向け：**Mode B（Strategic Partner）**

**判定不能時**：冒頭で立場確認を一度だけ行うか、両視点を分けて提示する。

### 1.3 断定と推測の分離（事故防止）
- `jobs_data.json` から算出した数値 → **事実**として提示可
- 外部データ → **出典と日付がある場合のみ**事実として提示可
- それ以外 → **推測（仮説）**として明記し、断定禁止

### 1.4 動的CV導線（Dynamic Conversion）
記事末尾には、テーマに関連する求人を**最大3件**提示する。
- 3件未満しか該当がない場合：不足分は「検索結果リンク」または「相談CTA」に置換する
- 求人は原則、以下条件で抽出する：
  - `country == ${Country}`
  - `job_category == ${JobCategory}`（または同義カテゴリ）
  - `status == active`（掲載中）
  - 直近更新優先

## 2. データ仕様（jobs_data.json）と必須フィールド

AIは **存在しないフィールドを仮定しない**。足りない場合は「不足項目」を列挙して生成を止める（下書き保存は可）。

### 2.1 最小必須フィールド（Minimum Schema）
- `job_id`（一意）
- `title`
- `country`（例：Thailand, Vietnam）
- `city`（任意だが推奨）
- `job_category`（例：Sales, Engineer, Admin）
- `salary_min`
- `salary_max`
- `salary_currency`（THB / VND / USD 等）
- `employment_type`（Local hire / Expat / Contract 等）
- `requirements`（配列 or 文字列）
- `description`
- `language_requirements`（例：Japanese N1, English B2）
- `updated_at`
- `status`（active / closed）
- `job_url`（記事でリンクするURL）

### 2.2 算出ルール（Salary Calculation Rules）
- 平均給与は **min/max の中点**を各求人で算出し、対象集合の平均を取る（例：`(min+max)/2`）。
- 上位20%給与レンジは、同じ手法で算出した中点を並べ、**80パーセンタイル以降**の平均を取る。
- 通貨が混在する場合は、**通貨を統一できない限り比較禁止**（比較するなら為替レートの出典と参照日が必須）。

## 3. 記事テンプレート（Article Templates）

以下の3タイプを基幹テンプレートとし、追加テンプレートを作る場合もこの思想（データ・直言・CV導線）を踏襲する。

---

### Type A：職種別・現実直視型（求職者ターゲット）
**ターゲット**：海外就職に幻想を持つ20〜30代（顕在〜準顕在）

**タイトル変数**：
- `${CountryJP}で${JobCategoryJP}として働く現実｜給与相場${Year}年版`

**構成（見出し固定）**
1. H1 タイトル
2. H2 まず結論（現実を一言で）
3. H2 給与相場（自社データ）
4. H2 生活シミュレーション（前提を明記）
5. H2 求人票が要求する「現実のスキル」
6. H2 失敗パターン（典型例・注記付き）
7. H2 それでも行くなら（具体行動）
8. H2 関連求人（最大3件）

**生成要件（AI指示）**
- 導入は必ず「幻想→否定」だが、人格攻撃は禁止（厳しいが論理的）。
- `jobs_data.json` から `${Country}` × `${JobCategory}` をフィルタし、以下を算出して提示：
  - 求人数（n）
  - 給与中点の平均
  - min/max の分布（可能なら四分位）
- 生活シミュレーションは、**前提（家賃・食費・税）を数値で置く場合**、許可された外部データの出典を必須とする。
  - 出典がない場合：シミュレーションは「レンジ（低/中/高）」で示し、断定しない。
- `requirements` から頻出語トップ3を抽出し、各項目について「求人側がそれを求める理由」を説明する。
- 結論は必ず CTA：
  - 関連求人3件 OR 検索結果リンク OR 相談CTA

---

### Type B：採用市場レポート型（企業ターゲット）
**ターゲット**：人が採れず困っている日系企業の採用責任者／現地拠点長

**タイトル変数**：
- `${CountryJP}の日系企業が${JobCategoryJP}採用に失敗する3つの理由`

**構成（見出し固定）**
1. H1 タイトル
2. H2 まず結論（失敗の核）
3. H2 市場の現実：給与レンジ（自社データ）
4. H2 失敗理由①（給与）
5. H2 失敗理由②（条件・運用）
6. H2 失敗理由③（意思決定の遅さ・権限）
7. H2 取るべき選択肢（2択を明示）
8. H2 相談導線（採用要件レビュー）

**生成要件（AI指示）**
- 「御社」など過度なへりくだりは禁止。外部顧問として事実を提示する。
- `jobs_data.json` から `${Country}` × `${JobCategory}` の給与分布を算出し、
  - 中央値
  - 上位20%平均（競合水準）
  - 求人数（n）
  を提示し「その条件で採れる確率」を言語化する。
- `description` から不人気条件（例：土曜出勤、長時間、転居頻繁、権限なし）を抽出し、**歩留まりへの影響**として説明する。
- 最後に必ず「二択」を迫る：
  - (A) 給与・条件を市場へ寄せる
  - (B) 要件・期待値を下げる（または日本人でなく現地人に切替）

---

### Type C：求人詳細拡張型（SEO特化）
**ターゲット**：特定求人の情報を比較している顕在層

**タイトル変数**：
- `${JobID}の募集要項を徹底解説｜${Title} in ${CountryJP}`

**構成（見出し固定）**
1. H1 タイトル
2. H2 この求人の結論（向く人／向かない人）
3. H2 仕事内容（求人票の翻訳ではなく意図を解説）
4. H2 求める人物像（requirementsの裏側）
5. H2 給与・待遇（前提明記）
6. H2 しんどいポイント（誤魔化さない）
7. H2 選考で見られる点（想定質問）
8. H2 応募前チェックリスト

**生成要件（AI指示）**
- 箇条書きの言い換えは禁止。「なぜそう書かれているか」を解説して文章化する。
- 事例は「典型例（合成）」として注記を入れる。
- **構造化データ（JobPosting）**を埋め込む。
  - 生成は必ず `jobs_data.json` の値から行い、欠損時は出力しない（エラーとして扱う）。

## 4. 変数定義（Variables Definition）

| 変数名 | 定義・ソース | 例 |
|---|---|---|
| `${Country}` | データ上の国キー | Thailand |
| `${CountryJP}` | 表示用（日本語） | タイ |
| `${JobCategory}` | データ上の職種キー | Sales |
| `${JobCategoryJP}` | 表示用（日本語） | 営業 |
| `${Year}` | 現在年 | 2026 |
| `${N}` | フィルタ後求人数 | 42 |
| `${AvgSalary}` | 給与中点平均 | 75,000 THB |
| `${MedianSalary}` | 中央値（中点） | 70,000 THB |
| `${Top20Salary}` | 上位20%平均（中点） | 95,000 THB |
| `${TopSkills}` | requirements頻出トップ3 | 英語, B2B営業, マネジメント |
| `${JobID}` | job_id | TH-12345 |
| `${Title}` | title | Sales Manager |

## 5. 内部リンク戦略（Internal Linking）

### 5.1 記事 → 求人検索（検索導線）
- 職種語が出たら、職種カテゴリ検索へリンクする。
- 国が出たら、国フィルタ検索へリンクする。

**リンク生成ルール**
- 固定の `q=sales` のような直書きは禁止。
- 可能ならサイト側の正式パラメータ仕様（例：`country=th&category=sales`）に合わせる。
- 不明な場合は、暫定でサイト内検索URL（/jobs）へのリンクに留める。

### 5.2 記事 → 記事（トピッククラスター）
- 国別ピラー（例：タイ就職の現実）→ 職種別（営業/IT/管理）→ 実務（ビザ/税）へ必ず相互リンク
- 同国・同職種の既存記事がある場合、**上位2本**へリンク（重複防止）

### 5.3 求人レコメンド（Dynamic Widget）
記事末尾に以下のプレースホルダーを設置：
```html
<div id="related-jobs" data-country="${Country}" data-category="${JobCategory}"></div>
```

## 6. 品質保証（QA）チェックリスト（公開前の必須）

**数値**
- [ ] 数値は `jobs_data.json` または許可データに基づく
- [ ] 通貨・期間・母数（n）が明記されている
- [ ] 推測は「推測」と明記されている

**事例**
- [ ] 実例の匿名化、または合成典型例で注記がある
- [ ] 「先週面談した」等、事実誤認させる表現がない

**法制度**（該当時）
- [ ] 国・更新日・一次ソースが明記されている
- [ ] 出典不明の断定がない

**CV導線**
- [ ] 関連求人が最大3件入っている（不足時の代替導線あり）
- [ ] Modeに一致したCTAになっている（求職者＝応募/相談、企業＝要件レビュー/相談）

## 7. AIへの実行指示テンプレート（運用コマンド）

### 7.1 Type A（求職者向け）
- 入力：`jobs_data.json`、`brand_voice.md`、本ガイド
- 指示例：

> `jobs_data.json` を読み込んでください。\
> 変数 `${Country}` = "Vietnam"、`${CountryJP}` = "ベトナム"、`${JobCategory}` = "Engineer"、`${JobCategoryJP}` = "ITエンジニア"、`${Year}` = "2026" として、\
> 本ガイドの **Type A** に基づき記事を執筆してください。\
> その際、`brand_voice.md` の **Mode A** を適用し、数値は `jobs_data.json` から算出したもののみ断定してください。\
> 生活費・税の数値を置く場合は、許可された外部データの出典と更新日を明記できる場合に限ります。

### 7.2 Type B（企業向け）
> `${Country}` と `${JobCategory}` を指定し、Type B に従って市場レンジ（中央値・上位20%）を算出し、企業が取るべき二択を提示してください。

### 7.3 Type C（求人拡張）
> `${JobID}` を指定し、該当求人データを参照して Type C を生成してください。JobPosting 構造化データは欠損がない場合のみ出力してください。

## 8. 付記（運用上の非交渉ルール）
- 量産のために信用を毀損しない。
- “それっぽい数字”を作るくらいなら、数字なしで書く。
- 迷ったら「出さない」を選ぶ（下書き保存に留める）。
